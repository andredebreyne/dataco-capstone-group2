# Q05 Geographic Global Map Dashboard Data

Page: Power BI geographic global-map view

## Objective

This support artifact prepares map-ready geographic data for the Power BI global
map page. It gives the dashboard author both text location fields and rounded
latitude/longitude fields so Power BI can use either geocoding or coordinate
placement.

The page should answer:

> Where are scored order items concentrated, and which locations show elevated
> delivery-risk or AO3 priority exposure?

## Governed Input

The geographic summary is generated by:

```text
src/dashboard/build_powerbi_geographic_summary.py
```

The script joins:

| Source | Purpose |
| --- | --- |
| `gold/ao3_risk_margin_segments` | Governed held-out AO3 scored order-item population. |
| `silver/dataco_customer_regional_features` | Decision-time destination geography, including normalized location text and rounded coordinates. |

The join uses:

```text
Order_Id
Order_Item_Id
order_date_DateOrders
```

The output Delta table defaults to:

```text
/Volumes/workspace/default/raw_data/gold/powerbi_geographic_summary
```

When the Databricks serving-layer registration runs, the table is published as:

```text
workspace.default.powerbi_geographic_summary
```

## Grain

One row per destination geography group:

```text
map_location_country
map_location_region
map_location_state
map_latitude
map_longitude
geo_coordinates_available
```

This is an aggregated dashboard table, not an order-level table.

## Power BI Fields

| Field | Recommended Power BI use |
| --- | --- |
| `map_location_label` | Tooltip or Location fallback. |
| `map_location_country` | Country slicer or map hierarchy. |
| `map_location_region` | Region slicer or hierarchy level. |
| `map_location_state` | State/location hierarchy level. |
| `map_latitude` | Latitude field when using coordinate-based maps. |
| `map_longitude` | Longitude field when using coordinate-based maps. |
| `geo_coordinates_available` | QA filter to compare coordinate-backed and text-only locations. |
| `order_item_count` | Bubble size or location volume. |
| `high_risk_order_rate` | Color saturation for AO1 delivery-risk exposure. |
| `avg_ao1_predicted_late_delivery_probability` | Tooltip or risk metric. |
| `avg_ao2_predicted_order_profit` | Tooltip for expected profitability context. |
| `avg_ao3_predicted_margin` | Tooltip for margin context. |
| `protect_high_value_at_risk_count` | Tooltip or operational-priority indicator. |
| `expedite_selectively_count` | Tooltip for high-risk low-margin exception volume. |

## Recommended Visual Setup

Use the Power BI map visual or Azure Maps visual with:

| Visual well | Field |
| --- | --- |
| Latitude | `map_latitude` |
| Longitude | `map_longitude` |
| Location | `map_location_label` |
| Size | `order_item_count` |
| Color | `high_risk_order_rate` |
| Tooltips | `map_location_country`, `map_location_region`, `map_location_state`, `avg_ao1_predicted_late_delivery_probability`, `avg_ao2_predicted_order_profit`, `protect_high_value_at_risk_count` |

For a coordinate-backed map, filter `geo_coordinates_available = 1`. For a
text-geocoded fallback, use `map_location_label` and location hierarchy fields.

## Guardrails

- Power BI must not use `Late_delivery_risk`, `Delivery_Status`,
  `Order_Profit_Per_Order`, actual shipping duration, or realized profit fields.
- The map summarizes predicted-score exposure. It does not show realized
  delivery performance.
- Coordinates are rounded and should be treated as approximate location support,
  not exact delivery addresses.
- If Power BI geocoding disagrees with coordinates, prefer the rounded latitude
  and longitude fields.

## Validation

After the build runs in Databricks, validate with:

```text
tests/data_validation/validate_powerbi_geographic_summary.py
```

The validator checks schema, target exclusion, coordinate ranges, nonnegative
counts, probability/rate bounds, metadata, and coordinate coverage.
